# API Documentation

This document describes the HTTP API exposed by the backend server.

The backend is a small Node.js HTTP server that supports:

- health checks
- URL analysis requests
- optional HTTP Basic Authentication
- optional static frontend serving

This document focuses on the **API surface** only.

---

## Base behavior

The server may be run in two common ways:

1. **API-only mode**
   - used during backend development
   - frontend runs separately

2. **Integrated mode**
   - used in production
   - backend serves the built frontend as well as the API

The API routes themselves remain the same in both cases.

---

## Content type

All API responses are JSON with:

```text id="w4e9d3"
Content-Type: application/json; charset=utf-8
```

Unless otherwise noted, API responses also include:

```text id="cc3v6e"
Cache-Control: no-store
```

---

## Request formatting

### Pretty printing

Some endpoints support the query parameter:

```text id="b3m1ti"
pretty=1
```

or

```text id="0m6v8p"
pretty=true
```

When enabled, JSON is formatted with indentation for easier manual inspection.

---

## Authentication

Authentication is optional and controlled by environment configuration.

When authentication is enabled:

- API routes require valid HTTP Basic Auth credentials
- static frontend routes also require valid credentials
- health routes may remain public if `AUTH_ALLOW_HEALTH=1`

If invalid or missing credentials are provided:

- the server responds with `401 Unauthorized`
- a `WWW-Authenticate` challenge is returned

If repeated failed authentication attempts occur and auth rate limiting is enabled:

- the server may respond with `429 Too Many Requests`

## Endpoints

### `GET /health`

### `GET /api/health`

Health check endpoint.

#### Purpose

Used for:

- deployment health checks
- container/platform liveness checks
- quick manual verification that the server is running

#### Response

**Status:** `200 OK`

Example:

```json id="y0jm1s"
{
  "ok": true,
  "service": "HubSpot Recommendation Tool API"
}
```

---

### `GET /analyze`

### `GET /api/analyze`

Analyzes a target website and returns recommendation-oriented output.

#### Query parameters

##### `url`

Required.

The absolute URL to analyze.

Example:

```text id="s7ul8d"
url=https://react.dev
```

##### `pretty`

Optional.

Use `1` or `true` to pretty-print the JSON response.

##### `includeMeta`

Optional.

Includes extra metadata when supported by the analysis route implementation.

Example request:

```http id="1h0hyq"
GET /api/analyze?url=https://react.dev&pretty=1&includeMeta=1
```

---

## Response behavior

The exact analysis payload is generated by the backend analysis pipeline and may include fields related to:

- normalized target URL
- detected technologies
- recommendation mappings
- confidence or match details
- metadata and warnings
- request or processing information

Because the analysis output is implementation-driven, downstream consumers should avoid assuming undocumented fields will never change.

When integrating with the frontend, prefer depending only on fields that are explicitly used by the UI.

## Common status codes

### `200 OK`

Returned for successful health checks and successful analysis requests.

---

### `400 Bad Request`

Returned when the request is invalid.

Typical reasons include:

- missing required query parameters
- invalid URL input
- malformed request values

---

### `401 Unauthorized`

Returned when authentication is enabled and the request does not include valid credentials.

Typical response characteristics:

- JSON error body may be returned depending on the auth path
- `WWW-Authenticate` header is included for the challenge flow

---

### `404 Not Found`

Returned when the request does not match a known API route.

Example:

```json id="cwq2sq"
{
  "ok": false,
  "error": "Not found"
}
```

---

### `429 Too Many Requests`

Returned when failed-auth rate limiting is enabled and the client has exceeded the configured threshold.

Example shape:

```json id="vtc6n0"
{
  "ok": false,
  "error": "Too many attempts. Please try again later.",
  "retryAfterSeconds": 60
}
```

The actual retry time depends on configuration.

---

### `500 Internal Server Error`

Returned for unexpected server-side failures.

In production, the error message is intentionally generic.

Example:

```json id="5l4aiq"
{
  "ok": false,
  "error": "Internal server error"
}
```

In non-production environments, the server may return a more specific error message to aid debugging.

## Headers

### Request headers

Common headers that may be used:

- `Accept`
- `Authorization`
- `X-Request-Id`

### Response headers

The server may include:

- `Content-Type`
- `Cache-Control`
- `X-Request-Id`
- `WWW-Authenticate`
- `Retry-After`

Security-related headers set by the server include:

- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: no-referrer`
- `X-Frame-Options: DENY`

CORS headers may also be included depending on configuration.

---

## CORS

CORS behavior is controlled by configuration.

Common patterns:

- `CORS_ALLOW_ORIGIN=*` for simple local development
- a specific allowed origin for tighter control
- `off` to disable CORS headers entirely

In production, when the backend serves the frontend from the same origin, permissive CORS is usually unnecessary.

---

## Static frontend serving

This backend can also serve a built frontend when configured to do so.

That behavior is **not part of the API**, but it affects deployment behavior:

- built frontend assets may be served directly
- SPA routes may fall back to `index.html`
- non-API browser navigation and API requests are handled differently

For deployment details, see:

- `README.md`
- `docs/ARCHITECTURE.md`
- `docs/OPERATIONS_GUIDE.md`

```bash id="b2vhzj"

```

---

## Operational notes

- keep health endpoints simple and stable
- avoid depending on undocumented analysis response fields in external integrations
- protect production deployments with HTTPS when using Basic Auth
- treat this API as a small, stable backend surface rather than a large public platform API
